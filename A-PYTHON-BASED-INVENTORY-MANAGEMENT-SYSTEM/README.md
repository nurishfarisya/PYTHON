1.1.	Add a new product
The adding new product function systematically manages the addition of new products to the inventory system.  When executed, the function first reads the products.txt file to determine the most recent product ID and automatically suggests the next available ID. By utilizing the zfill(3) function, the system maintains product IDs in a structured PXXX format, where P indicates a product and XXX represents a three-digit number (such as P001, P002). When users interact with the ID system, they can either press Enter to accept the automatically generated ID or input their own. The system also verifies the uniqueness of each ID by checking against existing entries in the products.txt file. If any validation rules are violated or if a duplicate ID is detected, the system provides clear error messages and continues to prompt for input until valid data is received.
For product details, each type of data undergoes specific validation processes. The name and description fields utilize the strip() function to remove any leading or trailing whitespaces, ensuring clean data entry. After stripping, the system verifies that these fields are not empty, as products must have meaningful identifiers and descriptions. Next, the Price validation incorporates additional complexity due to its numerical nature. The system employs a try-except block to catch any non-numeric inputs, ensuring that users can only enter valid numerical values. Beyond basic number validation, the system also verifies that the price is greater than zero, as negative or zero-priced products would not make sense in a real inventory system. The validation loop continues until a valid price is provided, with clear error messages guiding users toward correct input.
After successful validation of all required fields, the function writes the new product information to products.txt, in a new line. The data is stored in a structured format where each field is separated by pipe symbols (|), and the price is formatted to maintain two decimal places for consistency. The function concludes with a success message, confirming the addition of the new product to the inventory system.
Therefore, this adding new product function ensures that users enter valid product information.

1.2.	Update product details
The Product Details Update function systematically manages and modifies the details of existing products in the inventory system. Then executed, the function first reads all the existing products from the products.txt file and displaying them to the user. It then prompts the user to enter the Product ID of the product they wish to update. The input is validated to ensure it's not empty, and once a valid Product ID is provided, the function searches for a product with that ID. If a match is found, the function displays the current details of the product, including its name, description, and price.
After displaying the current product information, the user is prompted to provide new details. The product name, description, and price are all optional fields, meaning the user can leave them empty if they wish to keep the current information. If the user enters a new price, it is validated to ensure that it's a positive number and is formatted to two decimal places.    If the price input is invalid, the function will prompt the user to enter a valid value. Once the new details are gathered, the product information in the file is updated accordingly. The function then writes all the updated products back to the file, ensuring the changes are saved. If the Product ID provided by the user is not found, an error message is displayed, and the user is asked to enter a valid Product ID, allowing them to retry the process until a successful update occurs. 

1.3.	Add a new supplier
The adding new supplier function allows users to input suppliers into the system. This function runs with specific validation rules for each input field. The system reads the last line to get most recent id from supplier text file and automatically suggests the next available id. The zfill(3) function is used to maintain 3-digit format, such that 001, 002 while prefix ‘S’ is added for suppliers. The robust validation system ensures more proper ID format which is SXXX pattern. Error messages will be provided if the inputs are invalid or duplicated. The id validation is to ensure the uniqueness of each supplier, the code will continue to loop until valid input is received. 
Each data type has specific validation. Email requires symbols ‘@’ and ‘.’ to be included in email while phone numbers must match specific format (XXX-XXX-XXXX). The strict format checking for phone numbers including length must be exactly 12 characters, such that first section is 3 digits, followed by a hyphen, middle section must be 3 digits and followed by a hyphen, last section must include 4 digits. On the other hand, name validation uses strip() to remove any leading and trailing whitespaces. It is used to check if name in not empty after stripping. The data validation process helps to maintain consistent data format and prevent invalid data entry.
Therefore, this adding new supplier function ensures that users enter valid supplier ID, name, email address, and phone number, and preventing duplicated data.

1.5.	View inventory
The view inventory function is designed to provide users with a clear and organized view of all available products in the inventory. Its primary purpose is to read product information stored in a text file (product.txt) and display it in a structured format. The function starts by printing the column headers (Product ID, Product Name, Description, Price) to clarify the data being displayed. A separator line made of dashes follows to enhance readability and visually distinguish the headers from the product data.
Moving on, the function then attempts to open the file in read mode using try-except. By utilizing the with open syntax, it ensures that the file is properly closed after reading, even if an error occurs during the process. The contents of the file are read into a list using the readlines method, where each line represents a single product record to allow the function to handle files of varying lengths efficiently. If the file contains product records, the function iterates through each line, removes unnecessary whitespace as well as newline characters using strip() and prints the cleaned line to display the product details. This ensures the output is both accurate and visually consistent.
In cases where the file exists but contains no product records, the function gracefully handles the scenario by printing a message indicating that no products are found in the inventory. This prevents confusion and assures the user that the system is functioning correctly but simply lacks data. Additionally, if any errors occur while attempting to open or read the file, an IOError exception is caught and both the technical error message and a user-friendly message (“Cannot read from products file.”) are displayed. This is to ensure the program does not crash and provides useful feedback for troubleshooting.

The part for generation reports is used to show a clear and finalised data that saved in text files. It shows the data with a header and table with description. It allows user to select from a variety of reports. There are three types of reports such as low stock items report, product sales report and supplier orders report. For the low_stock_report() function determines whether products have a stock level below a specified threshold (such as lesser than 5) by checking the inventory’s current stock of each product. In a text file (product.txt), all the data that are relevant to low stock report are read from the file and displayed as a table. The table will show a clear header (Low Stock Items Report) first, and show the data according product ID, name, current stock and status of the product. 
Another report named product sales report is designed to generate the summary of each product’s total sales by examining orders from the orders.txt file. Based on the product prices, it determines the total number of sales and the sum of sold as well as presenting the data in a way that is straightforward. The differences between first report and this report, this report open two files to combined the useful data and show it at a table. This is to bring a readable result for user, so they no need opened two files to compared and check the stock. As usual, the table contain a header (Product Sales Report), and the data based on product ID, name, total sold and total sales (RM). 
The third type of report is called Suppliers Orders Report, this report display the quantity of each product that have been ordered by suppliers. The product.txt file and order.txt file provide most of the information. It helps the company monitor supplier demand by matching items to suppliers and calculating the total number of orders placed with each supplier. This report are generated depend on the suppliers ID, name of product and the total of orders. From this section, it lets users select the report they want to create and guarantees seamless navigation. The application is kept stable and accessible by including error handling to handle possible problems like missing files or wrong data types. 

1.7.	Main program
The main program serves as a central hub that integrates all functions together. It starts by initializing a running flag set to True, which controls the system's main loop. Inside this while loop, the program first displays the Inventory Management System menu, showing seven numbered choices. After displaying the menu, the system enters a try-except block to handle user input validation. Here, it prompts users to enter their choice and converts the input to an integer.
The system then uses a match-case structure to process the user's choice. Cases 1 through 6 call their respective functions (add product, update product, add supplier, place order, view inventory, and generate reports). Case 7 handles program exit by displaying a goodbye message and using break to exit the main loop. A default case handles invalid numeric inputs by displaying an error message. The try-except block catches ValueError exceptions for non-numeric inputs, displaying an appropriate error message.
After processing any user’s choice (except exit), the system enters a continuation check section. This section uses a while True loop to repeatedly display the menu and prompt for valid input, asking users if they wish to continue using the system. It accepts only 'y' or 'n' (case-insensitive) as valid inputs, displaying an error message for invalid entries. When users enter an invalid input, the system will ask again if they wish to continue or not. In the case where ‘y’ is selected, the system will display the menu again and the process of the main program will restart, but, If the user enters 'n', the running flag is set to False, ending the main loop. Finally, the system displays a farewell message before terminating. The entire system is started by calling main program at the end of the code.
